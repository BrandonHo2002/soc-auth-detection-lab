## Overview

This project demonstrates core security engineering and SOC monitoring concepts, with a focus on authentication systems, secure logging, and detection workflows using SIEM tooling.

The system simulates a real-world environment where authentication events are generated by an application and ingested into a SIEM (Splunk) for analysis and detection.

It includes:

* Secure user authentication with password hashing and MFA
* Account lockout and brute-force protection
* Centralized structured logging of security events
* Detection of suspicious activity using Splunk (SPL queries)

This project was built to simulate how a Security Operations Center (SOC) monitors authentication activity and investigates potential account compromise.

---

## Detection Logic

Detection is performed in Splunk by analyzing authentication logs generated by the application.

Example detections include:

* Multiple failed login attempts (brute-force behavior)
* Successful login after repeated failures
* MFA disable followed by login activity
* Logins from different IPs within a short time window
* Admin account activity anomalies

Splunk queries (SPL) are used to correlate events and identify suspicious patterns, simulating real SOC detection workflows.

## How It Works

### 1. Authentication System (Log Generation)

Run the authentication system to generate security events:

```bash
python auth_project/auth.py
```

This will produce structured authentication logs in:

```
auth_project/logs/pro_auth.log
```

---

### 2. Splunk (Detection Layer)

Authentication logs are ingested into Splunk, where detection logic is applied using SPL queries.

This allows:

* Event correlation across users and time windows
* Detection of suspicious authentication patterns
* Investigation of potential account compromise

---

### 3. Example Detections in Splunk

Below are example SPL queries used to detect suspicious activity:

**Brute-force detection:**

```spl
index=auth_logs
| stats count by user, event
| where event="LOGIN_FAIL" AND count > 5
```

Detects users with more than 5 failed login attempts, which may indicate a brute-force attack.

---

**Successful login after failures:**

```spl
index=auth_logs
| sort 0 _time
| streamstats count(eval(event="LOGIN_FAIL")) as failures by user
| where event="LOGIN_SUCCESS" AND failures > 3
```

Detects successful logins that occur after multiple failed attempts, which may indicate a compromised account.

---

**MFA disable followed by login:**

```spl
index=auth_logs
| sort 0 _time
| streamstats last(event) as prev_event by user
| where event="LOGIN_SUCCESS" AND prev_event="MFA_DISABLE"
```

Detects logins that occur after MFA has been disabled, which may indicate account takeover or security bypass.

## Limitations and Future Improvements

* Log ingestion is file-based rather than real-time streaming
* Detection rules are based on static thresholds
* No automated alerting pipeline (e.g., email, Slack, SOAR integration)
* No dashboard visualization (future improvement using Splunk dashboards)

## Purpose

This project was created to simulate real SOC workflows and demonstrate practical skills in:

* Authentication security
* Log generation and management
* SIEM-based detection (Splunk)
* Security event analysis

It is intended as a portfolio project for entry-level cybersecurity and SOC analyst roles.
