## Disclaimer

This project is intended for educational and portfolio purposes only. 
All authentication events, IP addresses, and attack scenarios are simulated within a controlled local environment.

## Overview

This project implements the core authentication system used to generate and analyze security events. It focuses on login handling, authentication workflows, logging, and detection logic for identifying suspicious authentication activity.

Detection and monitoring are primarily implemented in Splunk, while supplemental Python scripts demonstrate local detection logic and event correlation techniques for learning purposes.

The system simulates a real-world security monitoring setup where authentication events are generated by an application and ingested into a SIEM (Splunk) for monitoring and analysis.

It includes:

* Secure user authentication with password hashing and MFA
* Account lockout and brute-force protection
* Structured JSON logging of security events through a centralized logging module (`logger.py`)
* Detection of suspicious activity using Splunk (SIEM) with SPL queries

---

## Authentication Features

The authentication system includes:

* Secure password hashing using bcrypt with a pepper
* Account lockout after multiple failed login attempts (3 attempts → 5-minute lockout)
* Multi-Factor Authentication (MFA) using TOTP (PyOTP)
  * Admin users are required to configure MFA on first login
  * Regular users can optionally enable or disable MFA
  * MFA verification is enforced when enabled
  * For simulation purposes only, MFA setup may display QR codes or secret keys during onboarding flows. In production systems, MFA secrets would only be shown once and securely stored without re-display functionality.
* Role-based access control (admin vs user)
* Basic suspicious input detection and logging
* Structured JSON logging for all authentication events using a centralized logging module (`logger.py`) that standardizes event IDs, severity levels, statuses, and simulated source IPs
* Database interaction is handled via `auth_db.py`, which manages user storage, login tracking, and account updates in SQLite

---

## Detection Logic

Detection is primarily performed using Splunk Enterprise (trial environment), with supplemental Python-based detection scripts included for learning purposes.

Example detections include:

* Multiple failed login attempts (brute-force behaviour)
* Successful login after multiple failed attempts
* MFA disable followed by successful login (potential security bypass)
* Suspicious authentication patterns involving rapidly changing simulated source IPs
* Monitoring of privileged/admin account activity

SPL queries are used to correlate authentication events and identify suspicious patterns.

## How It Works

### 1. Authentication System (Log Generation)

Before running the system for the first time, create and activate a virtual environment:

Windows (PowerShell):

```bash
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

Install dependencies:

```bash
pip install -r requirements.txt
```

Then initialize the database:

```bash
python auth_project/init_db.py
```

(Optional but recommended) Create an admin account:

```bash
python auth_project/create_admin.py
```

Admin accounts are required to configure MFA on first login using a TOTP authenticator application (e.g., Google Authenticator).

Run the authentication system to start generating security events:

```bash
python auth_project/auth.py
```

(Optional) Run the local detection monitor in a separate terminal:

```bash
python auth_project/log_summary.py
```

User interactions such as login attempts, failed logins, and MFA verification generate structured authentication events.

These events are normalized through `logger.py` before being written to `logs/auth.log`.

This will produce structured authentication logs in:

```
logs/auth.log
```

---

### 2. Splunk (Detection Layer)

Authentication logs are ingested into Splunk Enterprise (trial environment), where they are analyzed using SPL queries to simulate SOC detection workflows.

Each log event includes fields such as event type, event ID, user, source IP, severity, and contextual details, enabling efficient filtering, correlation, and investigation.

This simulates how a Security Operations Center (SOC) monitors and analyzes authentication activity in a real-world environment.

---

### 3. Example Detections in Splunk

Below are example SPL queries used to detect suspicious activity:

**Brute-force detection:**

```spl
index=auth_project event="LOGIN_FAIL"
| stats count by user
| where count > 5
```

Detects users with excessive failed login attempts beyond the account lockout threshold. 

While the system locks accounts after 3 failed attempts, this detection identifies continued brute-force behaviour, such as repeated login attempts across multiple sessions or distributed attempts that persist despite account lockout protections.

---

**Successful login after multiple failed attempts:**

```spl
index=auth_project
| sort 0 _time
| streamstats count(eval(event="LOGIN_FAIL")) as failures by user
| where event="LOGIN_SUCCESS" AND failures > 3
```

Detects successful logins that occur after multiple failed attempts, which may indicate suspicious authentication activity or potential account compromise.

---

**MFA disable followed by login:**

```spl
index=auth_project
| sort 0 _time
| streamstats last(event) as prev_event by user
| where event="LOGIN_SUCCESS" AND prev_event="MFA_DISABLE"
```

Detects logins that occur after MFA has been disabled, which may indicate account takeover or security bypass.

## Splunk Dashboard

A Splunk dashboard was created to visualize authentication activity and support SOC-style monitoring and investigation.

The dashboard includes:

### 1. Failed Login Attempts Over Time
Displays login failures by user over time to identify brute-force behaviour and unusual spikes in activity.

### 2. Event Breakdown
Shows the distribution of authentication events (LOGIN_FAIL, LOGIN_SUCCESS, MFA_FAIL, MFA_SUCCESS, ACCOUNT_LOCK, etc.) to provide a high-level overview of system activity.

### 3. Top Attacked Users
Highlights the most targeted user accounts based on failed login attempts.

### 4. Top Attacker IPs
Identifies IP addresses responsible for the highest number of failed login attempts.

### 5. Suspicious Logins After Failures
Detects successful logins that occur after multiple failed attempts, indicating potential account compromise.

### 6. Account Lockouts
Displays accounts that have been locked due to repeated failed login attempts.

### 7. Suspicious Input Detection
Captures login attempts with invalid or potentially malicious input (e.g., special characters, SQL keywords, rejected inputs).

### 8. MFA Activity Monitoring
Tracks MFA success and failure events over time to identify abnormal authentication patterns.
